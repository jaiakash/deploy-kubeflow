## 1. What This Repo Does

This repo deploys the **Kubeflow Documentation AI Assistant** (docs-agent) on top of an OKE (Oracle Kubernetes Engine) cluster. The docs-agent is a RAG (Retrieval-Augmented Generation) chatbot that answers questions about Kubeflow by searching embedded documentation.

**What this repo does NOT do:**
- It does not write the application code (API server, pipelines). That lives in a separate repo.
- It does not create the OKE cluster itself. That is Akash's module. This repo assumes the cluster already exists and a kubeconfig has been written to disk.

**What this repo does:**
- Takes an existing OKE cluster
- Installs the Kubeflow platform (cert-manager, Istio, KFP, KServe, etc.)
- Deploys the docs-agent stack on top (Milvus, RBAC, API server, optional KServe LLM)

## 2. How Terraform Works — Quick Primer

If you are new to Terraform, here are the three concepts that matter most for this repo:

### Resources
A resource is a single piece of infrastructure Terraform manages. Examples:
```hcl
resource "kubernetes_namespace_v1" "docs_agent" { ... }   # one K8s namespace
resource "helm_release" "milvus" { ... }                   # one Helm release
```
Terraform tracks every resource in a **state file**. On the next `apply`, it compares the state file to the real world and only changes what has drifted.

### Modules
A module is a folder of `.tf` files that groups related resources. You call it from a parent:
```hcl
module "milvus" {
  source    = "./modules/milvus"
  namespace = "docs-agent"
}
```
Modules take **inputs** (variables) and expose **outputs** that other modules can reference. This is the primary way we wire one component to another.

### Providers
A provider is a plugin that knows how to talk to a specific API. We use three:
- `oracle/oci` — talks to Oracle Cloud
- `hashicorp/kubernetes` — talks to the Kubernetes API
- `hashicorp/helm` — talks to Helm (which talks to Kubernetes)

## 3. Root-Level Files Explained

### `providers.tf`

Declares which providers are needed and how to configure them.

```hcl
terraform {
  required_version = ">= 1.5.0"
  required_providers {
    oci        = { source = "oracle/oci",            version = ">= 5.0.0" }
    kubernetes = { source = "hashicorp/kubernetes",  version = ">= 2.25.0" }
    helm       = { source = "hashicorp/helm",        version = ">= 2.12.0" }
  }
}
```

**Why version floors (`>=`) and not exact pins:**
Exact pins go in `.terraform.lock.hcl` (auto-generated by `terraform init`). The floor in `providers.tf` says "any version at least this new". The lock file then records what was actually resolved, giving you reproducible installs without having to manually track exact patch versions here.

**Why the Helm provider uses `kubernetes = {}` (assignment, not block):**
Helm provider v3 changed the syntax. Prior versions used a nested block:
```hcl
# Old (provider < v3) — no longer valid
provider "helm" {
  kubernetes { config_path = "..." }
}
```
New:
```hcl
# Current (provider >= v3)
provider "helm" {
  kubernetes = { config_path = "..." }
}
```
Both the `kubernetes` and `helm` providers read the same kubeconfig file. This is the file Akash's OKE module generates via `oci ce cluster create-kubeconfig`.

### `backend.tf`

Tells Terraform where to store its state file.

```hcl
terraform {
  backend "s3" {
    endpoint = "https://<namespace>.compat.objectstorage.<region>.oraclecloud.com"
    bucket   = "terraform-state-docs-agent"
    key      = "docs-agent/terraform.tfstate"
    skip_region_validation      = true
    skip_credentials_validation = true
    skip_metadata_api_check     = true
    force_path_style            = true
  }
}
```

**Why S3 backend for OCI:**
Terraform has no native OCI backend. OCI Object Storage exposes an S3-compatible API, so the `s3` backend works — but only after disabling four AWS-specific validation checks (`skip_*` flags) that don't apply to OCI.

**Why `force_path_style = true`:**
OCI's S3-compat layer uses path-style URLs (`endpoint/bucket/key`), not virtual-hosted style (`bucket.endpoint/key`). Without this flag the SDK constructs the wrong URL and requests fail with DNS errors.

**Credentials:** Passed via env vars, never in code:
```bash
export AWS_ACCESS_KEY_ID="<OCI Customer Secret Key ID>"
export AWS_SECRET_ACCESS_KEY="<OCI Customer Secret Key value>"
```

---

### `variables.tf`

Single source of truth for every configurable value in the project. Grouped into five sections:

| Section | Variables | Purpose |
|---------|-----------|---------|
| OCI Auth | `tenancy_ocid`, `user_ocid`, `fingerprint`, `private_key_path`, `region` | OCI provider authentication |
| Cluster | `kubeconfig_path` | All K8s/Helm resources need to find the cluster |
| Kubeflow | `kf_version`, `kf_components` | Which branch and which components to install |
| Milvus | `milvus_namespace`, `milvus_release_name`, `milvus_chart_version` | Helm release identity; namespace shared by all docs-agent components |
| LLM | `deploy_kserve`, `external_llm_*`, `huggingface_token`, `kserve_model_id` | Toggle between external API and on-cluster GPU serving |
| API Server | `api_image`, `api_replicas` | Docs-agent FastAPI container |

**Key design choices:**
- `api_image` has **no default** — forces an explicit value. If you try to apply without building the image first, Terraform stops with a clear error rather than deploying a broken pod.
- `external_llm_api_key` and `huggingface_token` are `sensitive = true` — Terraform redacts them in all terminal output, plan diffs, and log files.
- `kf_components` is `list(string)` — the install script iterates this list. Adding or removing a component requires only a variable change, no HCL modification.

---

### `main.tf`

The root orchestration file. It calls all five modules in dependency order:

```
kubeflow_platform
    ├──► milvus         (depends_on: kubeflow_platform)
    ├──► rbac           (depends_on: milvus)
    ├──► api_server     (depends_on: milvus, rbac)
    └──► kserve_llm     (depends_on: kubeflow_platform, count = 0 by default)
```

**Why explicit `depends_on` between modules:**
Terraform automatically infers dependencies from variable references — if module B uses an output from module A, Terraform knows to create A first. But some dependencies are not captured by variable flow:
- `milvus` does not consume any output from `kubeflow_platform` in its variables (it only needs the namespace and Helm chart), but it does need Istio and cert-manager to be installed first or Helm will fail.
- Without `depends_on`, Terraform might try to deploy Milvus in parallel with Kubeflow — Helm would succeed but Istio wouldn't be there yet.

**The LLM toggle:**
```hcl
llm_endpoint = var.deploy_kserve ? module.kserve_llm[0].kserve_endpoint : var.external_llm_endpoint
```
One conditional at the `main.tf` level switches the API server between Groq and KServe. No changes needed inside any module. `module.kserve_llm[0]` uses index `[0]` because `kserve_llm` uses `count` (see below).

---

### `outputs.tf`

Declares what Terraform prints after `apply`. Outputs serve two purposes:
1. **Visibility** — engineers need the API URL, Milvus host, etc. immediately after deploy
2. **Composability** — another Terraform root (e.g., a monitoring stack) can read these values from the state file via `terraform_remote_state`

### `.gitignore`

| Pattern | What it excludes |
|---------|-----------------|
| `*.tfstate`, `*.tfstate.*` | State files — contain every resource's config including secrets |
| `*.tfvars` | Real variable values (OCIDs, API keys, tokens) |
| `!terraform.tfvars.example` | Un-excludes the template file so it is committed |
| `.terraform/` | Provider binaries (~100–200 MB) |
| `.terraform.lock.hcl` | Excluded during stub phase; add back once all modules are stable |
| `*.pem`, `*.key` | OCI API signing keys |
| `*_override.tf` | Local override files (used for test backend switching) |

---

### `terraform.tfvars.example`

A committed template showing every variable in a ready-to-fill format. Contributors copy this to `terraform.tfvars`, fill in real values, and can immediately run `terraform plan`.

The KServe block is commented out by default to reinforce that GPU mode is optional:
```hcl
# deploy_kserve     = true      ← uncomment to activate
# huggingface_token = "hf_..."
```

## 4. Module System — How Modules Connect

Every module under `modules/` has exactly three files:

```
modules/<name>/
├── main.tf       — the actual resources
├── variables.tf  — what the module accepts as input
└── outputs.tf    — what the module exposes to the caller
```

This is a Terraform convention enforced by our coding guidelines. It makes it easy to understand what a module does at a glance: read `variables.tf` for its contract, `outputs.tf` for what it produces, `main.tf` for how.

**How data flows between modules:**

```
variables.tf (root)
    │
    ▼
main.tf (root)
    │  passes vars to module calls
    ▼
modules/milvus/variables.tf        ← receives: namespace, release_name, chart_version
modules/milvus/main.tf             ← creates: kubernetes_namespace_v1, helm_release
modules/milvus/outputs.tf          ← exposes: milvus_host, milvus_port
    │
    ▼ (module.milvus.milvus_host passed to api_server module)
modules/api-server/variables.tf    ← receives: milvus_host, milvus_port, ...
```

## 5. Module Deep-Dives

### `kubeflow-platform` (Phase 6 — stub now)

**What it will do:** Wrap the official `kustomize build | kubectl apply` workflow in a `null_resource` with `local-exec` provisioners.

**Why not `kubernetes_manifest` for all 200+ KF resources:**
- Kubeflow is designed to be installed via Kustomize. Its manifests use `kustomization.yaml` files with overlays, patches, and generators that have no equivalent in the `kubernetes_manifest` resource.
- Tracking 200+ individual resources in Terraform state would create a maintenance nightmare — any upstream manifest change would require updating TF code.
- The `null_resource` approach runs the exact same install path as the official docs, reducing the chance of drift.

**Destroy provisioner:** A `local-exec` destroy provisioner runs `kubectl delete -k` in reverse component order. This is the main footgun with `null_resource` — Terraform has no visibility into what was actually created, so the destroy script must mirror the create script exactly.

---

### `milvus` (Phase 2 — COMPLETE)

**Resources:**
- `kubernetes_namespace_v1` — creates the `docs-agent` namespace
- `helm_release` — installs Milvus via the official Zilliz Helm chart

**Why standalone mode (not cluster):**
Milvus cluster mode requires ZooKeeper + distributed etcd + multiple Milvus pods. That easily consumes 8+ GB just for Milvus. Standalone packs everything into one pod with RocksMQ as the embedded message queue. The docs-agent doesn't need Milvus's distributed features — it has one collection and moderate query volume.

**Why `yamlencode({...})` for Helm values:**
Helm's `set {}` block uses dot-notation paths. The Istio annotation `sidecar.istio.io/inject` contains dots, which Helm interprets as nested object separators. Escaping them (`sidecar\.istio\.io/inject`) is fragile and hard to read. A single `values` block with `yamlencode` passes a proper YAML string to Helm — no escaping, nested objects work naturally.

**Why all three persistence layers are disabled:**

| Component | Why disabled |
|-----------|-------------|
| `etcd.persistence` | etcd stores Milvus schema — ephemeral is fine; recreates from app state |
| `minio.persistence` | MinIO stores segment files — ETL pipeline re-populates on restart |
| `standalone.persistence` | RocksMQ log data — if not disabled, chart creates a PVC with `helm.sh/resource-policy: keep`, which Helm leaves behind on uninstall, causing a 53-second delay in namespace termination |

**Why Istio sidecar is disabled in two places:**
1. `kubernetes_namespace_v1` label `istio-injection: disabled` — namespace-level control, affects all pods
2. `standalone.podAnnotations["sidecar.istio.io/inject"] = "false"` — pod-level override

The redundancy is intentional. If the namespace label is missed or overridden, the pod annotation is the last line of defense. Milvus's internal gRPC calls during startup fail if the sidecar intercepts them before Istio's mTLS certificates are ready.

---

### `rbac` (Phase 3 — stub)

**What it will do:** Create a `Role` and `RoleBinding` so Kubeflow Pipelines can discover the Milvus service.

**Why KFP needs this:**
Kubeflow Pipelines runs each pipeline step as a pod in the `kubeflow` namespace. When a pipeline step calls Milvus, it needs to resolve the service address. The `default-editor` service account (KFP's default runner) needs `get`/`list`/`watch` on `services` and `endpoints` in the `docs-agent` namespace.

Without this RBAC, the pipeline step can still reach Milvus by hardcoding the IP — but service discovery breaks and pods can fail if the Milvus pod restarts and the IP changes.

---

### `api-server` (Phase 4 — stub)

**What it will do:** Deploy the docs-agent FastAPI container as a `kubernetes_deployment` with a `kubernetes_service` and a `kubernetes_config_map`.

**Why ConfigMap for env vars (not hardcoded in the Deployment):**
The API server reads its configuration from environment variables (`MILVUS_HOST`, `MODEL`, `KSERVE_URL`, etc.). Putting them in a ConfigMap means:
- They're visible in the Terraform plan output (no surprises on apply)
- Changing an env var triggers a pod rollout automatically (Terraform detects the ConfigMap diff)
- Sensitive values (API keys) go in a `kubernetes_secret` instead

---

### `kserve-llm` (Phase 5 — stub)

**What it will do:** Deploy a KServe `ServingRuntime` + `InferenceService` for Llama 3.1-8B via vLLM.

**The `count` pattern:**
```hcl
module "kserve_llm" {
  source = "./modules/kserve-llm"
  count  = var.deploy_kserve ? 1 : 0
  ...
}
```
`count = 0` means Terraform creates zero instances of this module — nothing is applied. `count = 1` creates one instance. This is the standard Terraform pattern for optional singleton resources. The caller accesses it as `module.kserve_llm[0].kserve_endpoint` (with the `[0]` index).

**Why not a GPU node now:**
The OCI Always Free tier has no GPU. The module is written and validated (`terraform plan` with `deploy_kserve = false` confirms it's skipped), but the `count = 0` default means it costs nothing to have the code present.

## 6. Provider Architecture

```
┌─────────────────────────────────────────────────┐
│  Your machine running `terraform apply`         │
│                                                 │
│  ┌──────────┐  ┌────────────┐  ┌────────────┐  │
│  │ oci       │  │ kubernetes │  │ helm       │  │
│  │ provider  │  │ provider   │  │ provider   │  │
│  └─────┬────┘  └──────┬─────┘  └──────┬─────┘  │
└────────┼──────────────┼───────────────┼─────────┘
         │              │               │
         ▼              ▼               │
    OCI REST API   K8s API Server  ────┘
    (OCI Console)  (in OKE cluster)
                   (via kubeconfig)
```

- The `oci` provider is used only for Phase 6+ (OCI-level resources). For Phases 2–5, it is initialized but never invoked — which is why fake OCI credentials work when `-target=module.milvus`.
- Both `kubernetes` and `helm` providers connect to the same cluster via the same kubeconfig file. They're separate providers because Helm's abstraction level (Helm releases) is higher than raw Kubernetes manifests.

## 7. State Backend

```
Local machine                    OCI Object Storage
┌─────────────┐                  ┌──────────────────────────┐
│ terraform   │  S3-compat API   │ bucket: terraform-state   │
│ apply       │ ──────────────►  │ key: docs-agent/          │
│             │                  │      terraform.tfstate    │
└─────────────┘                  └──────────────────────────┘
```

The state file records every resource Terraform has created — its ID, attributes, and current configuration. Without remote state:
- Only one machine could run Terraform
- The state file could be lost with the laptop
- Two engineers running `apply` simultaneously would corrupt state

**S3-compat quirks for OCI:**
OCI Object Storage is not AWS S3. Four flags bypass AWS-specific checks that would otherwise fail:
```hcl
skip_region_validation      = true   # OCI regions don't match AWS region format
skip_credentials_validation = true   # OCI uses Customer Secret Keys, not IAM
skip_metadata_api_check     = true   # No AWS EC2 metadata endpoint on OCI
force_path_style            = true   # OCI uses path-style, not virtual-hosted
```

**Local testing override:**
Create `backend_override.tf` (gitignored via `*_override.tf`):
```hcl
terraform {
  backend "local" {}
}
```
Run `terraform init -reconfigure`. State is stored in `terraform.tfstate` locally. Delete this file to switch back to OCI.

## 8. Variable Flow — End to End

Here is how a single value (`milvus_namespace = "docs-agent"`) travels through the system:

```
terraform.tfvars
  milvus_namespace = "docs-agent"
         │
         ▼
variables.tf (root)
  variable "milvus_namespace" { default = "docs-agent" }
         │
         ▼
main.tf (root)
  module "milvus" {
    namespace = var.milvus_namespace    ← "docs-agent"
  }
         │
         ▼
modules/milvus/variables.tf
  variable "namespace" { type = string }  ← receives "docs-agent"
         │
         ▼
modules/milvus/main.tf
  resource "kubernetes_namespace_v1" "docs_agent" {
    metadata { name = var.namespace }    ← "docs-agent"
  }
  resource "helm_release" "milvus" {
    namespace = kubernetes_namespace_v1.docs_agent.metadata[0].name  ← "docs-agent"
  }
         │
         ▼
modules/milvus/outputs.tf
  output "milvus_host" {
    value = "${var.release_name}-milvus.${var.namespace}.svc.cluster.local"
           ← "my-release-milvus.docs-agent.svc.cluster.local"
  }
         │
         ▼
main.tf (root) — passes to api_server module
  module "api_server" {
    milvus_host = module.milvus.milvus_host
                ← "my-release-milvus.docs-agent.svc.cluster.local"
  }
```

Every value has a single definition point and flows down through the graph. Changing `milvus_namespace` in one place propagates everywhere automatically.

## 9. Dependency Chain — Why Order Matters

```
[Akash's OKE cluster] ─────────────────────────────────────────────┐
                                                                    │ kubeconfig written
                                                                    ▼
[kubeflow-platform]   Installs Istio, cert-manager, KFP, KServe ───┐
                      Without this, Milvus Helm hooks fail         │
                                                                    ▼
[milvus]              Creates docs-agent namespace + Helm release ──┐
                      Namespace must exist before RBAC              │
                                                                    ▼
[rbac]                Role + RoleBinding in docs-agent namespace ───┐
                      Must exist before API server starts           │
                                                                    ▼
[api-server]          Deployment reads Milvus host from module output
                      Starts only after Milvus is Running

[kserve-llm]          Independent of the above chain (count=0)
                      Activates when deploy_kserve = true + GPU node
```

**What breaks without this order:**
- Milvus before Kubeflow: Helm webhooks for cert-manager/Istio don't exist yet, install fails
- API server before RBAC: First request to Milvus returns 403, pod enters crash loop
- API server before Milvus: Connection refused on `19530`, pod enters crash loop

## 10. Local Testing Pattern

When testing a single module locally without OCI infrastructure:

**Step 1 — Override backend:**
```bash
cat > backend_override.tf << 'EOF'
terraform {
  backend "local" {}
}
EOF
terraform init -reconfigure
```

**Step 2 — Apply only the target module:**
```bash
terraform apply \
  -target=module.milvus \
  -var="tenancy_ocid=fake" \
  -var="user_ocid=fake" \
  -var="fingerprint=aa:bb:cc" \
  -var="api_image=placeholder" \
  -var="kubeconfig_path=$HOME/.kube/config" \
  -auto-approve
```
OCI credentials can be fake because `-target=module.milvus` only invokes the `kubernetes` and `helm` providers. The `oci` provider is initialized but makes no API calls.

**Step 3 — Verify:**
```bash
kubectl get pods -n docs-agent
kubectl port-forward svc/my-release-milvus -n docs-agent 19530:19530 &
python3 -c "from pymilvus import connections; connections.connect('default', host='localhost', port='19530'); print('OK')"
```

**Step 4 — Tear down:**
```bash
terraform destroy -target=module.milvus [same vars] -auto-approve
```

**Step 5 — Restore for production:**
```bash
rm backend_override.tf
terraform init -reconfigure  # switches back to OCI S3 backend
```

## 11. Common Pitfalls and How We Fixed Them

### Pitfall 1 — Provider v3 breaking changes

**Symptom:** `terraform validate` fails with "Unsupported block type" on the `helm` provider, and "Deprecated Resource" warning on `kubernetes_namespace`.

**Root cause:** Both `hashicorp/helm` and `hashicorp/kubernetes` released v3 with breaking changes:
- Helm: `kubernetes {}` block → `kubernetes = {}` assignment
- Kubernetes: `kubernetes_namespace` → `kubernetes_namespace_v1`

**Fix:**
```hcl
# providers.tf
provider "helm" {
  kubernetes = {              # was: kubernetes { ... }
    config_path = var.kubeconfig_path
  }
}

# modules/milvus/main.tf
resource "kubernetes_namespace_v1" "docs_agent" {   # was: kubernetes_namespace
```

---

### Pitfall 2 — S3 backend blocks local apply

**Symptom:** `terraform apply` fails with "Backend initialization required" even after `terraform init -backend=false`.

**Root cause:** `-backend=false` skips backend init but doesn't persist across `apply` calls. Terraform re-reads `backend.tf` on every command and sees the S3 backend, which isn't initialized.

**Fix:** Create `backend_override.tf` with `backend "local" {}`, run `terraform init -reconfigure`. The override file takes precedence over `backend.tf`. The `*_override.tf` pattern is already in `.gitignore`.

---

### Pitfall 3 — PVC left behind after destroy

**Symptom:** First `terraform destroy -target=module.milvus` prints:
```
Warning: These resources were kept due to the resource policy:
[PersistentVolumeClaim] my-release-milvus
```
Namespace deletion takes 53 seconds.

**Root cause:** The Milvus Helm chart sets `helm.sh/resource-policy: keep` on the standalone pod's PVC (for the RocksMQ data directory). Helm respects this annotation during `helm uninstall` and does not delete the PVC. Terraform then has to wait for the PVC to be garbage-collected when the namespace is deleted.

**Fix:** Add `standalone.persistence.enabled = false` to Helm values. No PVC is created, destroy is clean and fast (~5 seconds for Helm uninstall, ~43 seconds for namespace termination of remaining pods).

---

### Pitfall 4 — Istio mTLS conflicts with Milvus startup

**Symptom:** Milvus standalone pod enters `CrashLoopBackOff` when Istio is installed. Logs show gRPC connection failures to the internal etcd endpoint.

**Root cause:** Milvus standalone connects to its embedded etcd via gRPC during initialization. If the Istio sidecar is injected, it intercepts this traffic and applies mTLS. During early pod startup, Istio's certificates aren't ready yet, causing the gRPC handshake to fail.

**Fix (two layers):**
```hcl
# Namespace level — prevents sidecar injection on all pods in the namespace
resource "kubernetes_namespace_v1" "docs_agent" {
  metadata {
    labels = { "istio-injection" = "disabled" }
  }
}

# Pod level — belt-and-suspenders in case the namespace label is missed
standalone = {
  podAnnotations = { "sidecar.istio.io/inject" = "false" }
}
```
